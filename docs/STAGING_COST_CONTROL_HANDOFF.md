# Handoff — Staging cost control: hard-cap the twins so a soak can never double prod spend

**For the dual-repo session** (has both `ai-operations-command-center` and the trading-bot repo).
Prod LLM spend is ~$50/day. The staging twins exist ONLY as a development tester/debugger before
promotion — they must be **cheap by default and hard-capped by construction**, so a forgotten
weekend soak costs single-digit dollars, never a second $50/day.

Plan-first applies: state problem/approach/risks and get approval before code. Dev branches only.
CC dev branch: `claude/gracious-euler-dAdgn`.

---

## 1. Ground truth — CC's spend machinery (verified, with provenance)

- **Meter:** `runner/ledger/budget.py` — `record_spend()` writes
  `workspace/ledger/daily-spend.json`. Path is **checkout-relative**, so staging gets its **own
  fresh ledger** → by default a staging twin inherits a **full, separate $80/day allowance**.
- **Daytime cap:** `get_daily_cap()` (budget.py:42) reads
  `config/budgets.yaml → budgets.daily_limits.total_spend_limit_usd` (**$80.00**). No env
  override exists today.
- **⚠️ THE BIG ONE — off-hours lane is UNCAPPED by default:** `get_offhours_cap()`
  (budget.py:78) returns **infinity** unless `TONY_OFFHOURS_BUDGET_USD` is set, and
  `is_budget_exceeded(off_hours=True)` (budget.py:91) consults it when the market is closed
  (`runner/main.py:541`). A weekend/evening soak is almost ENTIRELY off-hours — i.e. the exact
  soak window runs in the lane with **no spend ceiling**. This must be capped in staging.
- **Gate sites:** `runner/main.py:541` (cycle budget gate), `:558` (per-pod), spend recorded at
  `runner/agents/base.py:306` → `record_spend`.
- **Why env, not YAML:** `config/budgets.yaml` is tracked; editing it in the staging checkout
  dirties the tree and **`promote_staging.sh` refuses to promote a dirty tree** (its guard at
  lines 31–33). Caps for staging must therefore arrive via `.env` → which needs the small code
  change in §2. Do NOT have the setup script edit YAML.

## 2. Task A (CC repo) — env overrides for the two caps

1. In `budget.py`, let env override the caps, default = current behavior (prod unchanged):
   - `get_daily_cap()`: honor `CC_DAILY_BUDGET_USD` if set/parseable, else budgets.yaml value.
   - (`get_offhours_cap()` already env-driven — no change.)
2. In `scripts/setup_staging.sh`, append to the staging `.env` block:
   ```
   # --- Staging spend caps: this twin is a tester, not a second producer -----
   CC_DAILY_BUDGET_USD=5            # hard daytime ceiling for the whole twin
   TONY_OFFHOURS_BUDGET_USD=3       # off-hours research lane (UNCAPPED in prod by design)
   ```
   Add both keys to the script's `OVERRIDE_KEYS` comment-out sweep. Pick final numbers with the
   operator; intent is "a full soak ≈ a couple of dollars."
3. Tests: cap override honored, garbage value falls back, default unchanged.
4. **Audit item:** verify every LLM call site in the runner flows through the budget gate /
   `record_spend` (e.g. `tony_synthesis` daily wrap, telegram chat replies, deep-dives). Any
   direct-API path that bypasses the meter is a hole in the cap — list what you find; fix only
   with approval.

## 3. Task B (bot repo) — same audit on the scanner twin

Enumerate the scanner's real-money API surfaces (LLM calls? paid data vendors? Finnhub tier/rate
limits?) with file:line provenance. For each: cap, disable, or document-as-free in the staging
`.env` generated by its `setup_staging.sh`. If the scanner has no metered LLM spend, say so
explicitly in its DEVELOPMENT.md (like the "no outbound senders exist" note).

## 4. Task C — two named soak modes (document in both DEVELOPMENT.mds)

- **PLUMBING SOAK (default):** caps above. Exercises scheduler/rollovers/sync/reconciler/
  bridge plumbing — the time-dependent bugs. LLM verdicts throttle early at the tiny cap; fine.
- **FULL-FIDELITY SOAK (explicit opt-in):** when the change under test IS the LLM path, operator
  raises `CC_DAILY_BUDGET_USD` (e.g. 15) for that soak only, then reverts. Never the default.

## 5. Task D (optional hardening) — auto-stop so "forgot to turn it off" can't bill for a week

Idle-but-running staging still cycles and spends up to its caps daily. Add an optional systemd
drop-in/timer in the setup scripts (commented in the printed instructions): e.g. a
`cc-staging-autostop.timer` that stops both staging services N hours after start (suggest 72h),
with the runbook note that a deliberate longer soak just restarts them. Keep it dead simple.
